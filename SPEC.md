# Длинные нарды — спецификация

Онлайн-приложение для игры в длинные нарды двух игроков по приглашению. Бэкенд на Go, фронтенд на TypeScript. Реал-тайм через WebSocket. Сервер — единственный источник правды.

---

## 1. Игровые правила (зафиксированный вариант)

### Доска и старт

- 24 пункта. Шашки двух цветов: **белые** и **чёрные**.
- У каждого игрока 15 шашек.
- Стартовая расстановка:
  - Белые: 15 на пункте **24** (голова белых).
  - Чёрные: 15 на пункте **12** (голова чёрных).

### Направление

Оба игрока движутся против часовой стрелки.

- Белые: `24 → 23 → … → 2 → 1 → выкид`.
- Чёрные: `12 → 11 → … → 1 → 24 → 23 → … → 14 → 13 → выкид`.

### Кубики

- Бросает **сервер** криптостойким ГСЧ.
- Обычный бросок — 2 хода (значения кубиков).
- Дубль — 4 хода одного и того же значения.
- Если хотя бы один пипс может быть использован — игрок обязан ходить. Если ни один не может — ход переходит сопернику.
- При двух возможных пипсах, если можно использовать только один, обязателен ход бо́льшим.

### Ход

- Шашка перемещается на ровно столько пунктов вперёд (по своему направлению), сколько указано пипсом.
- Можно ставить только на свою или пустую клетку. **На клетку противника ставить нельзя**.
- Битья нет.
- Шашка не «прыгает» — учитывается конечный пункт.

### Правило головы

За один ход с головы можно снять только одну шашку.

**Исключение:** на **первом ходу партии** при выпадении `6:6`, `4:4` или `3:3` разрешено снять с головы две шашки.

### Правило шести

Применяется **только на конец хода** (а не после каждого отдельного перемещения).

Запрет действует, если выполнены **оба** условия одновременно:
1. У соперника **ноль шашек в его доме** (в доме соперника, не в своём): для белого блокирующего — пункты 13–18, для чёрного — пункты 1–6.
2. В конце хода у блокирующего получилось **6 или более подряд** занятых пунктов.

Как только хотя бы одна шашка соперника зашла в его дом — ограничение снимается до конца партии.

Внутри хода временно держать 6 подряд можно; проверка применяется только к финальной позиции после END_TURN.

**Реализация:** `LegalMoves` фильтрует одиночные шаги по принципу «существует ли последовательность использования оставшихся пипсов, ведущая к легальной финальной позиции». Если у шага нет ни одного валидного продолжения — шаг отклоняется.

### Дом и сброс

- Дом белых: пункты **1–6**.
- Дом чёрных: пункты **13–18**.
- Выкидывать можно только когда **все 15 шашек** игрока — в его доме.
- Пипс выкидывает шашку с пункта, равного значению пипса (для белых: пипс 3 выкидывает шашку с пункта 3; для чёрных эквивалентно по своей шкале — пипс 3 выкидывает с пункта 16).
- Если в пункте, равном пипсу, шашек нет, пипс выкидывает шашку с **самого дальнего занятого** пункта меньшего номера в доме (по своему направлению). Перемещение внутри дома по этому пипсу также легально.

### Победа и очки

- **Оин** — 1 очко: проигравший успел выкинуть хотя бы одну шашку.
- **Марс** — 2 очка: проигравший не выкинул ни одной шашки.
- **Кокс** — 3 очка: марс, и при этом у проигравшего осталась шашка либо в доме победителя, либо на голове.

Партия одиночная (счёт не накапливается между играми) — это MVP.

### Право первого хода

- Каждый игрок кидает один кубик. У кого больше — ходит первым кубиками `(своё, чужое)`.
- При равенстве — перебрасывают.

## 2. Архитектура

```
/cmd/server                  — точка входа
/internal/domain             — чистая игровая логика (без I/O, 80% тестов)
/internal/game               — менеджер игровых сессий
/internal/transport/http     — REST: auth, приглашения, создание игры
/internal/transport/ws       — WebSocket: игровой канал
/internal/storage            — Postgres-репо (отложить, начать с in-memory)
/internal/auth               — токены/сессии
/internal/protocol           — типы сообщений WS, общие с клиентом
```

Принципы:
- `domain` не зависит ни от чего, кроме stdlib. Все правила нард — там.
- `game` оркеструет сессии: хранит `GameState`, принимает действия, рассылает обновления.
- `transport` — тонкий слой над `game`.
- Сервер хранит и валидирует **полное** состояние игры. Клиент получает state после каждого валидного действия.

## 3. Доменная модель (скетч на Go)

```go
type Color uint8
const (
    White Color = iota
    Black
)

// Point: 1..24 для обычных пунктов, 0 — выкид.
type Point uint8

// Board[i] — количество и цвет шашек на пункте i+1.
// Знак: >0 белые, <0 чёрные, 0 пусто. Модуль — количество.
type Board [24]int8

type Dice struct {
    A, B     uint8
    IsDouble bool
    Remaining []uint8 // пипсы, ещё не использованные в этом ходу
}

type Move struct {
    From Point // 1..24
    To   Point // 1..24, 0 = выкид
    Pip  uint8
}

type GameState struct {
    Board      Board
    Turn       Color
    Dice       Dice
    BorneOff   [2]uint8       // [White, Black]
    IsFirstMove [2]bool        // флаги «первого хода партии» для каждого
    Status     GameStatus     // WaitingForRoll, WaitingForMove, Finished
    Winner     Color          // если Finished
    WinKind    WinKind        // Oin, Mars, Koks
}

// Чистые функции
func InitialBoard() Board
func NextPoint(c Color, from Point, pip uint8) Point      // возвращает 0 при выкиде
func LegalMoves(s GameState) []Move
func Apply(s GameState, m Move) (GameState, error)
func IsTurnComplete(s GameState) bool
func Winner(s GameState) (Color, WinKind, bool)
func RollDice(r io.Reader) Dice                            // r = crypto/rand
```

Инварианты, которые проверяются в тестах:
- Сумма `|Board[i]|` всех белых + `BorneOff[White]` == 15. То же для чёрных.
- На пункте только один цвет (`Board[i]` не может быть «смешанной»).
- При `Status == WaitingForMove` все шашки игрока — в его доме перед сбросом.

## 4. Протокол WebSocket

Транспорт: JSON-сообщения. Поле `type` — discriminator.

### Клиент → сервер

| `type` | Payload | Когда |
|---|---|---|
| `JOIN` | `{ gameId, token }` | После принятия приглашения |
| `ROLL_FOR_FIRST` | `{}` | Определение первого хода |
| `ROLL` | `{}` | Начало своего хода |
| `MOVE` | `{ from, to }` | Каждый шаг хода |
| `END_TURN` | `{}` | Завершить ход (если ещё остались пипсы — отклоняется, если их можно использовать) |
| `RESIGN` | `{}` | Сдаться |

### Сервер → клиент

| `type` | Payload | Когда |
|---|---|---|
| `STATE` | `{ board, turn, dice, borneOff, status, isFirstMove }` | После каждого изменения |
| `LEGAL_MOVES` | `{ moves: [{from,to,pip}] }` | После ROLL и после каждого MOVE |
| `OPPONENT_JOINED` | `{ name }` | Соперник подключился |
| `OPPONENT_LEFT` | `{}` | Соперник отвалился |
| `GAME_OVER` | `{ winner, kind }` | Партия закончилась |
| `ERROR` | `{ code, message }` | Невалидное действие |

Правила протокола:
- `STATE` — полный снапшот, клиент не реконструирует частично.
- На каждое валидное `MOVE` сервер пересчитывает `LEGAL_MOVES` и шлёт обоим.
- На реконнект клиент получает `STATE` + `LEGAL_MOVES`.

## 5. Поток партии

1. Игрок A создаёт игру → получает `gameId` и ссылку-приглашение.
2. Игрок B открывает ссылку, авторизуется, делает `JOIN`.
3. Оба делают `ROLL_FOR_FIRST`. Сервер определяет, кто ходит.
4. Первый игрок: `ROLL` → сервер кидает кубики → присылает `STATE` + `LEGAL_MOVES`.
5. Игрок шлёт `MOVE` пока есть легальные ходы и пипсы. Каждый раз — `STATE` + `LEGAL_MOVES`.
6. Когда пипсов не осталось (или нет легальных ходов) — сервер сам делает `END_TURN`, передаёт ход сопернику.
7. Победа → `GAME_OVER`.

## 6. План TDD (порядок)

Каждый пункт — отдельный коммит: `red` → `green` → `refactor`.

### Прогресс

- ✅ Этап 1 (#1–#2): доска и расстановка
- ✅ Этап 2 (#3–#6): навигация
- ✅ Этап 3 (#7–#10): простой ход
- ✅ Этап 4 (#11–#12): дубль
- ✅ Этап 5 (#13–#17): правило головы
- ⏳ **Этап 6 (#18–#20): правило шести** — #18–#19 закрыты (`SixBlockAllowed`); #20 отложен до этапа 10 (требует `Apply`/`IsTurnComplete`)
- ⏹ Этапы 7–11 (#21–#37) — впереди

### Этап 1 — доска и расстановка
1. `InitialBoard()` — 15 на 24 для белых, 15 на 12 для чёрных, остальное пусто.
2. Инвариант: всегда 30 шашек суммарно.

### Этап 2 — навигация
3. `NextPoint(White, 24, 6) == 18`.
4. `NextPoint(Black, 12, 6) == 6`.
5. `NextPoint(Black, 5, 6)` пересекает «границу» → `23`.
6. `NextPoint(c, point, pip)` возвращает `0` (выкид), если выход за дом.

### Этап 3 — простой ход
7. Легальный ход на пустую клетку.
8. Легальный ход на свою.
9. Нелегальный ход на чужую.
10. Нелегальный ход не в свою сторону.

### Этап 4 — дубль
11. Дубль = 4 пипса.
12. Применение четырёх пипсов подряд.

### Этап 5 — правило головы
13. Только одна с головы за обычный ход.
14. Две с головы на первом ходу при `6:6`.
15. Две с головы на первом ходу при `4:4`, `3:3`.
16. Не две с головы при `5:5` на первом ходу.
17. Не две с головы при `6:6` на втором ходу.

### Этап 6 — правило шести (проверка только на конец хода)
18. Соперник 0 в его доме + финальная позиция с блоком 6+ у блокирующего → нелегально. Включая блоки, пересекающие границу 24↔1 для чёрных.
19. Соперник ≥1 в его доме → любой блок легален.
20. Промежуточный блок из 6 внутри хода (даже при сопернике 0 в доме) → легален, проверка только на END_TURN. **Покрывается интеграционным тестом на этапе 10 (Apply): сама `SixBlockAllowed` не знает контекста «промежуточный/финальный» — вызывающий код применяет её только к финалу.**

### Этап 7 — дом и сброс
21. Нельзя выкидывать, пока не все в доме.
22. Выкид точным пипсом с равного пункта.
23. Выкид «переборным» пипсом с самого дальнего пункта.
24. Перемещение внутри дома по легальному пипсу — допустимо.

### Этап 8 — конец игры
25. Победа = у победителя `BorneOff == 15`.
26. Оин: у проигравшего `BorneOff >= 1`.
27. Марс: у проигравшего `BorneOff == 0`, шашек в доме победителя/на голове **нет**.
28. Кокс: у проигравшего `BorneOff == 0`, есть шашка в доме победителя или на голове.

### Этап 9 — обязательные ходы
29. Если можно использовать только один пипс из двух — обязателен бо́льший.
30. Если ни один пипс не используется — ход переходит.

### Этап 10 — оркестрация
31. `Apply` обновляет `Remaining`.
32. `IsTurnComplete` срабатывает, когда `Remaining` пуст или нет легальных ходов.

### Этап 11 — транспорт и сессии (после доменного ядра)
33. WS handshake + `JOIN`.
34. Полный поток `ROLL`-`MOVE`-`END_TURN` с двумя клиентами в памяти.
35. Реконнект.
36. Persistence через Postgres.
37. Аутентификация.

## 8. Безопасность (краткий чек-лист)

- Только WSS/HTTPS в проде.
- Кубики бросает сервер через `crypto/rand`.
- Все ходы валидируются сервером (полный пересчёт `LegalMoves`).
- Авторизация в WS-хендшейке (токен в заголовке/cookie, не в URL).
- Rate limit на REST.
- Логирование всех ходов для разбора жалоб.
