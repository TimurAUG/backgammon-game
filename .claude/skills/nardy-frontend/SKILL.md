# Фронтенд backgammon-game — архитектура и TDD

Источник правды для структуры `/web/`, потока данных и TDD-дисциплины фронта. При расхождении с кодом — прав этот файл и [`FRONTEND_SPEC.md`](../../../FRONTEND_SPEC.md).

Связанные скиллы:
- `nardy-svelte` — конвенции Svelte 5 (runes, реактивность, тесты компонентов).
- `nardy-protocol` — WS-сообщения (типы клиентских/серверных).
- `nardy-rules` — правила игры (для unit-тестов, если когда-нибудь дублируем валидации на клиенте — пока не дублируем).

## Структура

```
web/
  src/
    main.ts              — bootstrap
    App.svelte           — root, выбирает Connect или Game
    protocol/messages.ts — TS-зеркало internal/protocol/messages.go
    transport/ws.ts      — WSClient (connect/send/onMessage/реконнект)
    stores/
      game.ts            — board/turn/dice/borneOff/status/isFirstMove/legalMoves/gameOver
      connection.ts      — idle | connecting | connected | reconnecting | error
    screens/             — Connect, Game, GameOver
    components/          — Board, Checker, Dice, ActionBar, GameOver
    lib/geometry.ts      — координаты SVG для пункта/шашки
  tests/                 — общие фикстуры, mock WebSocket
```

Полная карта файлов и обязанностей — `FRONTEND_SPEC.md` § 2.

## Поток данных

```
WebSocket → WSClient → messages.ts (parse) → gameStore
                                                  ↓
                                           UI (Svelte reactivity)
                                                  ↓
UI event → gameStore action → WSClient.send → WebSocket
```

**Жёсткое правило**: сторы держат только то, что прислал сервер. Никаких локальных правил игры — все валидации на бекенде. Подсветка легальных ходов — фильтр `legalMoves` из последнего `LEGAL_MOVES`. Если в UI хочется «проверить ход локально перед отправкой» — отказать, ждать `ERROR` от сервера.

## TDD-дисциплина для фронта

**Никакого production-кода в `/web/src/` без падающего теста**, ссылающегося на конкретный пункт плана из `FRONTEND_SPEC.md` § 7.

### Цикл на каждый пункт плана

1. **Red.** Найти следующий пункт в `FRONTEND_SPEC.md` § 7. Написать тест в `*.test.ts` (для логики) или `*.test.svelte.ts` (для компонента) рядом с целевым файлом. `npm test` должен упасть с осмысленным сообщением. Коммит: `test: #<N> <описание>`.

2. **Green.** Минимальный код, чтобы тест прошёл. Никакой логики сверх покрытой тестом. Коммит: `feat: #<N> <описание>`.

3. **Refactor.** Если есть что чистить — почистить, тесты остаются зелёными. Коммит: `refactor: #<N> <описание>`. Пропустить, если рефакторить нечего.

### Как идти по плану

Строго по порядку этапов 0 → 1 → ... → 8 из `FRONTEND_SPEC.md` § 7. Не прыгать вперёд.

Внутри этапа с 3+ пунктами — группировать по парам (2 коммита на пару: `test:` + `feat:`). Так же, как делали на бекенде для этапов 5/6/7.

Большой пункт, затрагивающий >2 файлов — разбивать на под-циклы `a`/`b`/`c` (как `#34`/`#36` на бекенде).

## Конвенции тестов

- **Тестовый рантайм**: Vitest + jsdom.
- **Компонентные тесты**: `@testing-library/svelte`. Файлы: `Component.test.svelte.ts` рядом с `Component.svelte`.
- **Юнит-тесты** (stores, transport, lib, protocol): `*.test.ts` рядом с источником.
- **Mock WebSocket**: один шаблон в `web/tests/mockWebSocket.ts`, не плодить локально по тестам.
- **Фикстуры STATE**: фабрики типа `stateFixture({ turn: 'white', ... })` в `web/tests/fixtures.ts` — не «магические» JSON-объекты по тестам.
- **Имена**: `<юнит>_<сценарий>_<ожидание>` (например, `gameStore_onSTATE_updatesBoard`).
- **Сообщения assert**: читаются без открытия исходника.
- **Один сценарий на тест.** Никаких «и/и/и».
- **Указать пункт плана** в комментарии: `// FRONTEND_SPEC #7`.

## Что НЕ тестировать

- Сам WebSocket-runtime браузера. Тестируется WSClient через mock.
- CSS / точный layout. Тестируется наличие и атрибуты SVG-узлов, не пиксели.
- Правила игры. Они на бекенде; тестировать их на клиенте — дублирование с риском расхождения.
- Реальную сеть в юнит-тестах. Только мок.

## Антипаттерны

- Тест, который пройдёт даже если выкинуть реализацию.
- `if/else` внутри теста — это два теста.
- Импорт внутренних деталей `.svelte`-компонента вместо взаимодействия через DOM (`getByRole`, `findByText`).
- Хранение в сторе того, что не пришло из STATE («оптимистическое» применение MOVE — нет, ждём STATE).

## Когда отступить от TDD

- Этап 0 (каркас) — glue: `package.json`, конфиги Vite/TS/Svelte/ESLint, базовый `App.svelte` с заглушкой. Один коммит без теста.
- Чисто механический рефакторинг (переименование).
- Эксперимент в отдельной ветке, который выкидывается.

Во всех остальных случаях — сначала падающий тест.

## Чек-лист перед коммитом теста

- Один сценарий.
- Имя объясняет, что проверяется.
- Указан пункт плана.
- Фикстуры через фабрику, не «магические» объекты.
- Не использован реальный WebSocket / реальная сеть.
- Для компонента — взаимодействие через DOM-запросы, не через приватные методы.
