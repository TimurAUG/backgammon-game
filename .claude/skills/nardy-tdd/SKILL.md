# TDD-дисциплина для backgammon-game

## Жёсткое правило

**Никакого production-кода без падающего теста**, ссылающегося на конкретный пункт плана из SPEC.md → раздел 6 «План TDD».

## Цикл на каждый пункт плана

1. **Red.** Открыть SPEC.md, найти следующий пункт плана. Написать тест в `*_test.go` рядом с целевым файлом. Запустить `go test ./...` — тест должен упасть с осмысленным сообщением. Закоммитить как:

   ```
   test: #<N> <короткое описание>
   ```

2. **Green.** Написать **минимальный** код, чтобы тест прошёл. Никакой логики сверх той, что покрыта тестом. Если хочется добавить «заодно» — это следующий пункт плана, делается отдельным циклом. Закоммитить как:

   ```
   feat: #<N> <короткое описание>
   ```

3. **Refactor.** При необходимости почистить код. Тесты должны оставаться зелёными. Пропустить, если рефакторить нечего. Если делается — закоммитить как:

   ```
   refactor: #<N> <короткое описание>
   ```

## Какой тест следующий

Идти по пунктам SPEC.md → раздел 6 **строго по порядку**. Не прыгать вперёд.

Если очередной пункт оказался уже покрыт тестом, отметить в SPEC.md (зачеркнуть или пометить ✓) и взять следующий.

## Конвенции тестов

- Файлы: `*_test.go` рядом с тестируемым.
- Пакет: тот же, что у тестируемого (white-box), пока домен не стабилизировался.
- Фреймворк: `testing` + `github.com/stretchr/testify/require` (`assert` не используем — `require` останавливает тест после первого падения, тесты читаются проще).
- Имена: `TestXxx_Yyy_Zzz` — `Test<Func>_<Scenario>_<Expectation>`.
- Таблицы: для разных вариантов одного правила — table-driven с `t.Run(tc.name, ...)`.
- В комментарии к тесту указать пункт плана: `// TDD plan #18`.

## Что НЕ тестировать на доменном уровне

- Сеть, БД, время, логирование.
- Конкретный ГСЧ. Принимать `io.Reader` (или `func() uint8`) для броска, в тесте подавать `bytes.Reader` с фиксированными байтами.

## Чек-лист перед коммитом теста

- Один сценарий — не «и/и/и».
- Имя теста объясняет, что проверяется.
- Сообщение об ошибке читается без открытия исходника.
- Состояние доски в setup строится через хелпер `boardWith(...)` или фабрику, а не из «магического» массива.
- Указан номер пункта плана в комментарии.

## Антипаттерны

- Тестирование приватных деталей реализации вместо поведения.
- Тест, который пройдёт даже если выкинуть всю реализацию.
- Тест с условной логикой (`if/else` внутри теста) — это два теста.
- Mock’и в доменном слое. Домен — чистые функции.

## Когда отступить от TDD

- Чисто механический рефакторинг без изменения поведения (например, переименование).
- Эксперимент / прототип в отдельной ветке, который потом выкидывается.

Во всех остальных случаях — сначала падающий тест.
