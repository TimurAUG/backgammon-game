# WebSocket-протокол backgammon-game

Источник правды для всех сообщений между клиентом и сервером. При расхождении с кодом — прав этот файл.

Все сообщения — JSON с обязательным дискриминатором `type`.

## Клиент → сервер

### `JOIN`

Подключение к партии. Шлётся первым после открытия WS.

```json
{ "type": "JOIN", "gameId": "string", "token": "string" }
```

Токен может прийти и в `Authorization: Bearer`-заголовке handshake-запроса (путь для не-браузерных клиентов; заголовок имеет приоритет над `token` в payload). Браузерный `WebSocket` заголовки ставить не умеет, поэтому веб-клиент шлёт токен только в JOIN. Если токена нет ни там, ни там — сервер отвечает `ERROR { code: "UNAUTHORIZED" }`.

### `ROLL_FOR_FIRST`

Бросок одного кубика для определения первого хода.

```json
{ "type": "ROLL_FOR_FIRST" }
```

### `ROLL`

Бросок кубиков на свой ход. Допустим только при `status == "waitingForRoll"` и `turn == свой цвет`.

```json
{ "type": "ROLL" }
```

### `MOVE`

Один шаг шашки. Допустим только при `status == "waitingForMove"` и `turn == свой цвет`.

```json
{ "type": "MOVE", "from": 1-24, "to": 0-24 }
```

- `to == 0` означает выкид.

### `END_TURN`

Передать ход сопернику. Если есть пипсы, которые можно ещё использовать — сервер отклоняет с `MUST_USE_PIP`.

```json
{ "type": "END_TURN" }
```

### `RESIGN`

Сдаться.

```json
{ "type": "RESIGN" }
```

## Сервер → клиент

### `STATE`

Полный снапшот состояния. Шлётся **после каждого** валидного действия любого игрока.

```json
{
  "type": "STATE",
  "board": [int8 x 24],
  "turn": "white" | "black",
  "dice": {
    "a": 1-6,
    "b": 1-6,
    "isDouble": false,
    "remaining": [int]
  },
  "borneOff": { "white": 0-15, "black": 0-15 },
  "status": "waitingForRoll" | "waitingForMove" | "finished",
  "isFirstMove": { "white": true, "black": true }
}
```

Кодировка `board[i]`: знак — цвет (`>0` белые, `<0` чёрные, `0` пусто), модуль — количество.

### `LEGAL_MOVES`

Список легальных одиночных шагов для текущего игрока с учётом оставшихся пипсов. Шлётся **только** игроку, чей ход.

```json
{
  "type": "LEGAL_MOVES",
  "moves": [
    { "from": 1-24, "to": 0-24, "pip": 1-6 }
  ]
}
```

Если массив пуст — у игрока нет легальных продолжений, сервер автоматически передаст ход (через внутренний END_TURN).

### `OPPONENT_JOINED` / `OPPONENT_LEFT`

```json
{ "type": "OPPONENT_JOINED", "name": "string" }
{ "type": "OPPONENT_LEFT" }
```

### `GAME_OVER`

```json
{ "type": "GAME_OVER", "winner": "white" | "black", "kind": "oin" | "mars" | "koks" }
```

### `ERROR`

Невалидное действие. Партия не меняется.

```json
{ "type": "ERROR", "code": "string", "message": "string" }
```

## Коды ошибок

| `code` | Когда |
|---|---|
| `INVALID_MOVE` | Базовое нарушение: направление, чужая клетка, нет такого пипса. |
| `HEAD_RULE` | Попытка снять вторую с головы вне исключения для дублей 6:6/4:4/3:3 на первом ходу. |
| `RULE_OF_SIX` | Финальная позиция хода создаёт запрещённый блок (соперник 0 в его доме + блок 6+). |
| `MUST_USE_LARGER` | При двух пипсах надо использовать бо́льший. |
| `MUST_USE_PIP` | Попытка END_TURN, когда есть используемые пипсы. |
| `NOT_YOUR_TURN` | Действие не в свою очередь. |
| `INVALID_STATE` | Действие не подходит к `status` (например, MOVE при waitingForRoll). |
| `GAME_NOT_FOUND` | `gameId` неизвестен. |
| `UNAUTHORIZED` | Невалидный или истёкший `token`. |

## Инварианты протокола

- После любого валидного действия сервер шлёт:
  1. `STATE` — обоим клиентам;
  2. `LEGAL_MOVES` — только игроку, чей ход (если `status == "waitingForMove"`).
- Клиент **не реконструирует** состояние частично. Только из `STATE`.
- На реконнект клиент шлёт `JOIN` с тем же `gameId`/`token` и получает текущий `STATE` (+ `LEGAL_MOVES` если его ход).
- Сервер игнорирует команды от игрока, чей `turn != свой цвет` (отвечает `NOT_YOUR_TURN`).
- Поле `isFirstMove[color]` сбрасывается в `false` после первого END_TURN этого цвета.

## Версионирование

Пока без явной версии. При первом несовместимом изменении ввести поле `protocolVersion` в `STATE` и серверный handshake.
